# 热点刀锋 v4.1 升级方法论

> 2026-06-20 实战总结：8+技能生态的系统性升级方法。

## 升级策略

### 1. 先扫描再动手
升级前完整审查所有子技能+脚本+参考文档，列出不一致清单。本次扫描发现：
- 禁用语清单在5处维护，版本各不同
- v4.1升级方案在SKILL.md中重复3遍
- 14个Pitfalls有多个重复表达同一担忧
- retired-phrase-scanner.py停留在v3.4，与banned-words.md不同步

### 2. 单一信源原则
多处维护同一份数据必然导致不一致。解法：
- 创建 `references/banned-phrases-unified.md`（人类可读）+ `banned-phrases-data.json`（机器可读）
- 所有其他技能引用此文件，不再各自维护
- 扫描脚本从JSON读取规则，0维护成本

### 3. 脚本化验证 > LLM判断
写作质量验证、去重、禁用语检查、数字校验——这些机械操作不应该交给LLM。
每类检查写一个Python脚本，0 token 0误判。

### 4. 分级模型策略
| 任务类型 | 推荐模型 | 理由 |
|---------|---------|------|
| 评分/筛选/去重 | 轻量模型（MiMo） | 成本低，逻辑简单 |
| 创意/写作/润色 | 强模型（DeepSeek） | 质量优先 |
| 用户手动触发 | 默认模型 | 尊重用户选择 |

后置润色治标不治本——强模型写初稿才是正解。

### 5. P3观察项的正确处理
P3 Scrapling评估结论是"不需要"——curl_cffi覆盖全部5平台。
但仍然创建了HTML降级模板作为保险。
正确做法：评估结论+升级路径+备用方案，三者缺一不可。

## 关键教训

1. **SKILL.md瘦身效果显著**：283行→195行（-31%），LLM加载时注意力更集中
2. **curl_cffi不是万能的**：36氪gateway API对impersonate有兼容问题，需要保留legacy curl
3. **自动fallback比手动切换可靠**：curl_cffi失败时自动切legacy，不需要人工干预
4. **去重必须是独立步骤**：弱模型在prompt中说"去重"时经常跳过，必须用脚本硬匹配
